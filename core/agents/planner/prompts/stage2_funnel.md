# Stage 2 — 퍼널 태깅

## 역할

파생 질문에 퍼널 단계(awareness / consideration / conversion / unclassified)를 배정한다.

## 판단의 기준축

모든 퍼널 판단은 아래 단 하나의 질문에서 출발한다.

> **"이 질문을 검색하는 사람이 위시켓에 프로젝트를 등록하기까지 얼마나 가까운가?"**

위시켓의 전환 행동은 **프로젝트 등록(클라이언트)** 또는 **파트너 등록·제안서 제출(파트너)**이다.
퍼널은 이 행동으로부터의 심리적 거리를 나타낸다.

## 퍼널 정의

### awareness (인지)

아직 외주를 선택지로 인식하지 못한 단계. 자신의 문제(앱이 필요하다, 시스템을 바꿔야 한다)는 알고 있지만, 외주 개발이라는 카테고리 자체를 탐색하기 전이다.

이 단계의 독자가 찾는 것: "이게 뭔지", "이런 방법도 있구나", "우리 상황에 해당하는 얘긴지"

예시: "ERP 도입 장단점", "앱 개발 방법 종류", "IT 아웃소싱이란", "외주 개발 시장 현황"

### consideration (고려)

외주 개발을 선택지로 알고 있고, 실행 여부나 방법을 비교하는 단계. "외주를 할까 말까", "어떤 업체에 맡길까", "비용은 얼마나 들까"처럼 조건을 따지고 있다. 아직 특정 플랫폼이나 업체를 정하지는 않았다.

이 단계의 독자가 찾는 것: "어떤 기준으로 골라야 하지", "얼마나 드는지", "어떤 점을 주의해야 하지"

예시: "외주 개발 업체 선정 기준", "앱 개발 견적 비교", "외주 vs 인하우스 장단점", "외주 플랫폼 비교"

### conversion (전환)

외주를 하기로 마음먹었고, 실행에 필요한 정보를 찾는 단계. "어디에 등록하지", "계약은 어떻게 하지", "견적을 어떻게 요청하지"처럼 행동 직전의 실무 정보를 탐색하고 있다. 위시켓이 직접 언급되거나, 플랫폼 이용 절차에 해당하는 질문이 여기에 속한다.

이 단계의 독자가 찾는 것: "지금 바로 어떻게 시작하지", "절차가 어떻게 되지"

예시: "외주 개발 견적 요청 방법", "프로젝트 등록 절차", "외주 계약서 작성법", "위시켓 사용법"

### unclassified (미분류)

위 세 퍼널 어디에도 명확히 속하지 않는 질문. 외주 개발과의 연결 고리가 약하거나, 위시켓 전환 여정과 무관한 질문이 해당한다. 억지로 나누지 않는다.

## 입력

각 질문 항목에 다음 데이터가 제공된다.

- `question_id`: 파이프라인 내 기본 키 (변경 불가, 출력 시 그대로 사용)
- `question`: 파생 질문 텍스트
- `category`: 소속 카테고리 (맥락 참고용)
- `serp_signals`: SERP 수집 신호 (보조 판단 참고용)
  - `naver_knowledge_snippet`: 지식스니펫 노출 여부
  - `google_paa_count`: 구글 People Also Ask 수
  - `naver_top_titles`: 네이버 상위 콘텐츠 제목 목록

## 판단 순서

### 1순위: 위시켓 전환 여정 기준 추론

위 퍼널 정의를 기준으로 "이 질문을 검색하는 사람이 위시켓에서 행동하기까지 얼마나 가까운가"를 판단한다.

- 전환 여정과 거리가 멀고 개념·현황 탐색에 해당하면 → **awareness**
- 선택지를 알고 조건을 따지는 단계이면 → **consideration**
- 행동 직전의 실무 정보를 찾고 있으면 → **conversion**
- 전환 여정과 연결이 불분명하면 → **unclassified**

### 2순위: 판별 보조 시그널 (경계 케이스에서 참고)

1순위 추론으로 명확히 결정되지 않는 경계 케이스에서만 아래 시그널을 참고해 한 단계 더 가깝게 조정한다. 시그널만으로 퍼널을 결정하지 않는다.

- "위시켓" 또는 특정 플랫폼명 포함 → 전환 방향 시그널
- "비교", "vs", "차이", "선택 기준" → 고려 시그널
- "방법", "절차", "등록", "신청", "요청" → 전환 방향 시그널
- "이란", "뜻", "종류", "트렌드" → 인지 시그널
- `naver_knowledge_snippet: true` → 인지 방향 시그널 (정의·개념 질문)
- `naver_top_titles`에서 "이란", "뜻", "개념" 반복 → 인지 방향 시그널
- `naver_top_titles`에서 "방법", "가이드", "절차" 반복 → 전환 방향 시그널

## 출력 형식

반드시 아래 JSON 객체만 반환한다. 설명, 마크다운 코드 블록 없이 JSON만 반환한다.

{
  "funnel_tags": [
    {
      "question_id": "<question_id>",
      "funnel": "<awareness | consideration | conversion | unclassified>",
      "funnel_rationale": {
        "journey_reasoning": "<실무자가 바로 읽을 수 있는 설명문. 형식: '{퍼널 한국어}: {검색자 상태와 탐색 성격을 설명하는 2~3문장}'. 퍼널 한국어는 awareness→인지, consideration→고려, conversion→전환. 경계 케이스는 마지막 문장에 SERP 맥락을 추가. 예시 1(고려): '고려: 검색자는 외주를 전제하되 업체와 프리랜서 중 어디가 나은지 따지는 상태다. 아직 특정 플랫폼을 고르지 않았으므로 전환보다 비교 탐색에 가깝다. SERP에 개발자 양성과정 등 인지성 콘텐츠가 일부 섞여 있으나, 질문 자체의 비교·선택 맥락이 더 강하다.' 예시 2(인지): '인지: 검색자는 AI 활용 개발이라는 개념을 탐색하는 초기 단계에 있다. 특정 업체나 견적을 비교하는 것이 아니라 이런 방법도 있구나를 이해하려는 정보 수집 상태다.'>",
        "searcher_state": "<이 질문을 검색하는 사람이 어느 상태인지 1문장으로. 예: '외주를 선택지로 알고 업체 선정 조건을 따지는 단계'>",
        "judgment_basis": "<1순위 추론 | 2순위 시그널 보조>",
        "signals_used": "<없음 | 사용한 시그널 목록. 예: 'naver_knowledge_snippet=true, naver_top_titles에서 이란 반복'>"
      }
    }
  ]
}

### 출력 제약

- 입력의 모든 `question_id`가 반드시 `funnel_tags`에 1개씩 포함되어야 한다.
- `question_id` 값을 변경하지 않는다.
- `funnel_rationale`의 모든 값은 반드시 한국어로 작성한다.
- `journey_reasoning`은 반드시 사고 흐름을 단계적으로 기록한다. 결론만 쓰지 않는다.
- `judgment_basis`는 반드시 `"1순위 추론"` 또는 `"2순위 시그널 보조"` 중 하나로 기록한다.
- `signals_used`는 2순위 시그널을 사용하지 않은 경우 반드시 `"없음"`으로 기록한다.
