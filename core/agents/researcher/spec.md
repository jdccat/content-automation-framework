# 리서처 에이전트 — 작업 정의 v2 (수집 전담)

## 입력

메인 키워드와 고객 진입 모먼트를 전략 수립 단계에서 전달받는다.

## 사전 컨텍스트

별도 파일(context.yaml)에 정의된 항목을 참조한다. 리서처는 매 실행 시 전체 파일을 읽지 않고, 각 수집 단계에서 필요한 영역만 꺼내서 로드한다. 해당 영역이 비어 있으면 그 영역을 참조하는 수집 로직은 건너뛴다. 파일 구조와 항목 정의는 사전 컨텍스트 명세를 참조한다.

## 리서치 아카이브

리서처는 매 실행의 최종 산출물을 실행 날짜 키와 함께 아카이브에 저장한다. 아카이브는 두 가지 용도로 사용한다. 첫째, 1단계에서 신규 클러스터와 기존 아카이브 클러스터 간 의미 유사도를 비교하여 병합·신규·중복 플래그를 판단한다. 둘째, 이전 실행에서 클러스터에 포함되지 못한 고립 키워드를 다음 실행의 키워드 확장에 추가 시드로 투입할 수 있다.

---

## 1단계: 키워드 확장 및 클러스터링

사용자 입력에서 키워드 시드를 추출하고, 플랫폼별로 최대한 넓게 확장한 뒤, 의미 기반으로 클러스터링한다.

### 1-1. 사용자 입력 파싱

사용자가 입력한 질문에서 키워드 시드를 최대한 많이 추출한다.

추출 대상:

- 명사구 단위 키워드
- 질문에 내포된 의도 (정보 탐색 / 비교 / 문제 해결 등)
- 동의어, 유사 표현으로 변환 가능한 키워드
- 복합 질문인 경우 각 주제를 개별 키워드 시드로 분리

### 1-2. 플랫폼별 키워드 확장

추출된 키워드 시드별로, 각 플랫폼에서 독립적으로 확장을 수행한다. 플랫폼 간 수집은 병렬로 실행한다.

#### 구글 검색 수집

- 자동완성 키워드 (Google Autocomplete)
- 연관 검색어 (Related Searches)
- People Also Ask(PAA) 질문 목록
- 3단어 이상의 구체적 롱테일 조합까지 적극 확장

#### 네이버 검색 수집

- 자동완성 키워드 (네이버 자동완성 API)
- 연관 검색어 (네이버 연관 검색어)

### 1-3. 내부 고객 언어 대조 \[context: customer_language 존재 시에만 수행\]

영업 FAQ, 고객 지원 티켓 분류, 사이트 내부 검색어 로그를 참조해서, 검색 도구에서 나오지 않지만 실제 고객이 반복적으로 사용하는 표현을 추가 수집한다. 내부 데이터에서 발견된 키워드는 발견 출처를 "내부 데이터"로 표기한다.

### 1-4. 키워드 정리

#### 클러스터링

수집된 전체 키워드 목록을 LLM에 입력하고, 의미적으로 같은 검색 의도를 공유하는 키워드끼리 묶어 클러스터를 생성한다.

클러스터링 판단 기준:

- 사용자가 해당 키워드를 검색했을 때 기대하는 정보가 실질적으로 동일한가
- 하나의 콘텐츠로 두 키워드의 검색 의도를 모두 충족할 수 있는가

어떤 클러스터에도 의미적으로 속하지 않는 키워드는 고립 키워드로 분류한다.

#### 대표 키워드 선정

각 클러스터에서 LLM이 대표 키워드를 선정한다.

선정 기준: 해당 클러스터 내 하부 키워드를 가장 많이 포괄하는 키워드. 즉, 그 키워드 하나로 콘텐츠를 작성했을 때 클러스터 내 다른 키워드의 검색 의도까지 자연스럽게 커버할 수 있는 키워드를 선택한다.

동률 처리: 포괄 범위가 동일하다고 판단되면 키워드 길이가 짧은 쪽을 선택한다.

#### 아카이브 클러스터 비교

신규 생성된 클러스터의 대표 키워드를 기존 아카이브 클러스터 DB의 대표 키워드와 비교한다.

비교 방식: LLM이 신규 대표 키워드와 아카이브 대표 키워드 간 의미 유사도를 판단한다. 동일한 검색 의도를 다른 표현으로 말하고 있는지(예: "외주 개발" ↔ "아웃소싱 개발")를 기준으로 판단한다.

판단 결과:

- 병합: 의미적으로 동일한 기존 클러스터가 있으면 신규 키워드를 해당 클러스터에 병합하고, 대표 키워드는 포괄 범위가 더 넓은 쪽으로 갱신한다.
- 신규 생성: 기존 클러스터와 의미적으로 구분되면 신규 클러스터로 등록한다.
- 중복 플래그: 기존 클러스터와 완전히 겹치는 경우 중복 위험으로 플래그한다. 이 플래그는 이후 전략 수립 시 중복 콘텐츠 발행 방지에 사용된다.

### 1-5. 포커스 클러스터 선정

아카이브 비교까지 정리된 전체 클러스터 중에서, 사용자 입력 질문 관련 콘텐츠를 제작할 때 필요한 정보를 수집하기에 적합한 클러스터만 선별한다.

선정 기준: LLM이 각 클러스터에 대해 "이 클러스터의 키워드가 해당 질문 관련 콘텐츠를 제작할 때 필요한 정보를 수집하기에 적합한 검색 키워드인가"를 판단한다. 답이 Yes인 클러스터만 포커스 클러스터로 선정한다.

입력: 사용자 원본 질문 목록 + 전체 클러스터(대표 키워드, 하부 키워드)
출력: 포커스 클러스터 ID 목록

### 1단계 아웃풋

1단계 완료 후 두 가지 경로로 결과를 전달한다.

**아카이브 업데이트**: 아카이브 비교에서 확정된 전체 클러스터(병합 / 신규 / 중복 플래그 포함)를 아카이브 DB에 반영한다. 포커스 여부와 무관하게 모든 클러스터가 아카이브에 저장된다.

**2단계 전달**: 포커스 클러스터 목록(대표 키워드, 하부 키워드)만 2단계로 넘긴다.

1단계 아웃풋 필드:

- 클러스터 ID, 대표 키워드, 선정 근거, 포커스 여부
- 포함 키워드 목록 (키워드별 발견 출처: 구글 / 네이버 / 키워드 도구 / 내부 데이터)
- 아카이브 비교 결과 (병합 / 신규 / 중복 플래그)
- 고립 키워드 목록

---

## 2단계: 검증 수집

1단계에서 선정된 포커스 클러스터별로 공략 가치를 판단하는 데 필요한 정량 데이터를 수집한다. 클러스터별 수집은 병렬로 수행하며, 각 클러스터 내에서도 플랫폼별 수집을 병렬로 실행한다.

### 실행 구조

```
포커스 클러스터 A ──┬── 구글 수집 ──┐
                    └── 네이버 수집 ──┤── 클러스터 A 통합
                                      │
포커스 클러스터 B ──┬── 구글 수집 ──┤
                    └── 네이버 수집 ──┤── 클러스터 B 통합
                                      │
  ...                                 └── 2단계 아웃풋 합산
```

### 2-1. 검색량 수집

클러스터 내 각 키워드의 월간 검색량을 플랫폼별로 수집한다.

#### 구글

- 월간 검색량
- 클러스터 단위 합산 검색량
- 최근 6개월 검색량 추이 (상승 / 안정 / 하락)

#### 네이버

- 월간 검색량 (PC/모바일 분리)
- 클러스터 단위 합산 검색량
- 최근 6개월 검색량 추이 (상승 / 안정 / 하락)

### 2-2. 상위 콘텐츠 분석

클러스터 대표 키워드로 검색했을 때의 상위 10개 결과를 플랫폼별로 수집한다.

#### 구글 상위 10개

각 결과에 대해:

- 제목, URL
- H2 헤딩 구조
- 발행일
- 콘텐츠 유형 (기업 블로그 / 미디어 / 가이드 / 비교 리뷰 / 사례 연구 / 공식 문서 / 기타)
- \[context: competitors\] 경쟁사 플래그

#### 네이버 상위 10개

각 결과에 대해:

- 제목, URL
- 노출 영역 (통합검색 VIEW / 블로그 탭 / 지식iN 등)
- 발행일
- 콘텐츠 유형 (블로그 / 카페 / 지식iN / 뉴스 / 공식 사이트 / 기타)
- \[context: competitors\] 경쟁사 플래그

### 2-3. SERP 피처 수집

콘텐츠 전략에 직접 영향을 주는 SERP 특수 요소만 수집한다.

#### 구글

- AI 오버뷰(AI Overview) 존재 여부
- 추천 스니펫(Featured Snippet) 존재 여부 및 소스 URL
- People Also Ask 질문 목록

#### 네이버

- 지식스니펫 노출 여부
- 스마트블록 노출 여부 및 구성 요소

### 2단계 아웃풋

- 클러스터별 합산 검색량 (구글 / 네이버 분리, PC/모바일 분리, 6개월 추이 포함)
- 상위 10개 콘텐츠 메타 시트 — 구글 / 네이버 분리 (제목, URL, H2 구조, 유형, 발행일, \[해당 시\] 경쟁사 플래그)
- 구글 상위 콘텐츠 H2 주제 목록 + 1단계 PAA 질문 목록 (병렬 나열)
- SERP 피처 체크리스트 — 구글 / 네이버 분리

---

## 3단계: 외부 AI 환경 수집

클러스터가 현재 AI 검색 환경에서 어떻게 다뤄지고 있는지 확인하기 위한 데이터를 수집한다.

### 3-1. GEO 인용 현황 수집

클러스터 대표 키워드를 질문 형태로 변환해서 브라우저를 통해 다음 4개 AI 서비스에 각각 입력한다.

대상 서비스:

- ChatGPT
- Perplexity
- Claude
- Gemini

각 서비스별 수집 항목:

- 응답에서 인용된 소스 URL 목록
- 위시켓 블로그 도메인 포함 여부
- 인용된 경쟁사 도메인 목록
- 각 인용 소스가 어떤 맥락에서 인용되었는지 짧은 요약 문장

### 3단계 아웃풋

- GEO 인용 소스 목록 — 서비스별 분리 (ChatGPT / Perplexity / Claude / Gemini)
  - URL, 위시켓 포함 여부, 경쟁사 도메인 목록
- 인용 소스별 인용 맥락 요약

---

## 아카이브 저장

실행 완료 후, 최종 산출물 전체를 실행 날짜 키와 함께 아카이브에 기록한다.

---

## 최종 산출물 구조

리서처 에이전트는 세 단계의 아웃풋을 클러스터 단위로 합쳐서 플래너 에이전트에 전달한다. 최종 산출물은 사전 정의된 JSON 스키마에 맞춰서 출력한다.

하나의 클러스터 단위에는 다음이 포함된다:

### 식별 정보

- 클러스터 ID, 대표 키워드, 선정 근거, 포커스 여부

### 확장 데이터 (1단계)

- 포함 키워드 목록 (키워드별 발견 출처: 구글 / 네이버 / 키워드 도구 / 내부 데이터)
- 아카이브 비교 결과 (병합 / 신규 / 중복 플래그)

### 검증 데이터 (2단계)

- 합산 검색량 (구글 / 네이버 분리, PC/모바일 분리, 6개월 추이)
- 상위 콘텐츠 메타 시트 — 구글 / 네이버 분리 (제목, URL, H2 구조, 유형, 발행일, \[해당 시\] 경쟁사 플래그)
- H2 주제 목록, PAA 질문 목록
- SERP 피처 체크리스트 — 구글 / 네이버 분리

### 외부 AI 환경 데이터 (3단계)

- GEO 인용 소스 목록 — 서비스별 분리 (ChatGPT / Perplexity / Claude / Gemini, 위시켓 포함 여부, 경쟁사 도메인 목록, 인용 맥락 요약)

### 부속 데이터

- 고립 키워드 목록 (클러스터 미소속, 아카이브 시드용)

이 시트에는 판단이나 우선순위가 없다. 의도 분류, 퍼널 추정, 문제 진술 추출, 콘텐츠 갭 판정, 기존 콘텐츠 대조는 모두 플래너 에이전트가 수행한다.
