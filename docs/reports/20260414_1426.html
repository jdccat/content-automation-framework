🔧AI 응답 교정 (ChatGPT Key Points)
ChatGPT 응답: '프리랜서(개인) vs 외주 개발사(SI) 비교 제시: 비용 vs 안정성 트레이드오프' — 프리랜서가 비용 면에서 유리한 것처럼 구조화
비용은 방식(프리랜서/개발사)이 아니라 요구사항 범위와 파트너에 따라 결정된다. 프리랜서 월 단가 400~600만 원대도 있고, 소형 개발사가 오히려 저렴한 경우도 있다. '프리랜서=싸다, 개발사=비싸다' 프레이밍은 금지(outsourcing_industry_context.md §1-7 참조). 방식 비교가 필요하면 같은 요구사항 기준 개별 사례로 범위감만 전달해야 한다.
Claude 응답: '외주 개발 성패는 기획 단계에서 80% 결정된다' — 수치 단정
출처 불명의 수치 단정. '기획이 중요하다'는 사실이지만 '80%'라는 수치는 근거가 없다. data_evidence.md §2 채택 기준(조사 주체·구체 수치·2024년 이후)을 충족하지 못하므로 인용 불가. '기획 준비만 잘해도 프로젝트 완성도가 크게 달라진다'(outsourcing_industry_context.md §2) 수준으로만 서술.
AI 응답 3개 모두: '요구사항 정의서(RFP) 작성'을 첫 단계로 제시하고 독자가 직접 작성할 것을 권유
tech_foggy 독자는 구조화된 요구사항 문서를 직접 작성하지 못한다(product_target.md §2 tech_foggy 권유 금지 목록). '나열은 하지만 빠진 게 뭔지 모름' 상태다. RFP 작성을 권유하는 대신, '핵심 기능을 일상어로 나열 + 참고 서비스 URL 정리' 수준으로 낮추고, 나머지는 매니저 상담으로 연결하는 구조가 적절하다.
공통 주제: '계약 체결 시 작업 범위(SOW)·소유권·유지보수 조건을 명확히 정의해야 한다' — 클라이언트가 직접 계약 조항을 정의할 것을 권유
계약 조항 직접 설계는 비개발자 클라이언트가 할 수 없는 영역이다(outsourcing_industry_context.md §1-4). 대신 '위시켓 표준계약서에는 하자보수·지재권·대금보호가 포함되어 있다'는 해결 구조를 보여주는 것이 올바르다. 독자가 해야 할 것은 계약서 설계가 아니라 '지재권 양도 명시 여부 확인'처럼 단순 확인 수준으로 낮춰야 한다.
경쟁글 공통 주제: '프리랜서·개발사·플랫폼' 3가지 경로를 나란히 제시하며 위시켓·크몽·이랜서를 병렬로 언급
경쟁사(크몽, 이랜서 등)를 직접 언급하는 것은 wishket_service_context.md §6 경쟁사 관련 금지 사항. 비교가 필요한 맥락에서는 '일반적인 프리랜서 플랫폼', '단순 매칭 서비스' 수준으로만 언급 가능. 차별화는 매니저 동반·대금보호·표준계약서 팩트로 자연스럽게 서술.
🚧집필 가드레일 (Don'ts)
✖ 아이디어만 있으면 바로 업체부터 찾아도 된다고 제시하지 않는다
→ 핵심 기능 목록과 참고 서비스 URL을 먼저 정리하는 것이 첫 단계. 기획이 어려우면 기획 단계부터 도움을 요청하는 것도 방법이다
✖ '개발사가 기획을 도와줄 것'처럼 기획 책임을 파트너에게 돌리지 않는다
→ 기획이 어려우면 기획부터 별도 의뢰하는 방법이 있음을 명시하고, 위시켓에서 매니저 상담을 통해 요구사항 정리를 도움받을 수 있음을 연결
✖ '외주를 맡기면 알아서 다 해준다'는 수동적 역할 인식을 심지 않는다
→ 외주 프로젝트도 클라이언트의 적극적 참여(중간 확인, 피드백)가 필요하다는 점을 명확히 안내
✖ '소스코드는 당연히 받을 수 있다'고 전제하지 않는다
→ 계약 시 '소스코드를 포함한 지재권 일체' 양도를 명시해야 한다고 안내
✖ '서버·도메인은 개발사가 세팅해줄 것'이라는 기대를 심지 않는다
→ 서버·도메인·앱스토어 계정은 클라이언트 회사 명의로 만들어달라고 요청해야 한다고 안내
✖ '하자보수 기간에 기능 개선도 요청할 수 있다'고 안내하지 않는다
→ 하자보수(버그 무상)와 유지보수(기능 추가·운영 유상)를 명확히 구분해서 안내
✖ '프리랜서가 싸다', '개발사가 비싸다' 등 방식별 비용 고저를 단정하지 않는다
→ 방식별 비교가 필요하면 같은 요구사항 기준의 개별 사례로 범위감만 전달
✖ '예산 N만 원이면 이 방식', 'N만 원 이상이면 저 방식'처럼 예산 구간별 방식을 분기하지 않는다
→ 독자의 상황(경험, 역량, 긴급도)을 기준으로 분기. 예산은 결과적 제약으로만 언급
✖ '아키텍처를 물어보세요', '기술 스택을 확인하세요' 같은 비개발자가 실행 불가능한 행동을 제안하지 않는다
→ 비슷한 프로젝트 경험이 있는지, 결과물을 보여줄 수 있는지 확인하도록 안내
✖ '이 6가지 체크리스트를 모두 확인하세요' 식의 과도한 숙제를 주지 않는다
→ 핵심 2~3가지만 + '나머지는 매니저와 함께' 구조로 연결

같은 '웹서비스'라도 어떤 기능이 얼마나 포함되어 있느냐에 따라 예산이 크게 달라지는 것이 정상이다. 단순한 단일 기능 서비스와 결제·회원 관리·데이터 연동이 모두 들어간 서비스는 같은 '웹서비스'라는 이름이지만 구현 공수가 전혀 다르다. 예산 차이의 원인은 대부분 포함 기능의 범위 차이이므로, 견적서를 비교할 때 금액 전에 어떤 기능이 포함되어 있는지를 먼저 보는 것이 순서다.
예산은 기능 범위의 결과이므로 절대적인 정답은 없지만, 실제 프로젝트 사례를 보면 대략적인 범위감을 잡는 데 도움이 된다. 단순한 단일 기능 웹서비스는 수백만 원대에서 시작하고, 비교·연산·회원 관리 등이 포함된 중형 서비스는 수천만 원대로 올라간다. 같은 예산 구간의 사례라도 포함된 기능이 다르므로, '이 사례와 같은 예산이면 된다'고 단정하기보다 '우리 서비스는 이 사례보다 기능이 많으니 더 높을 수 있다'는 방식으로 가늠하는 것이 현실적이다.

💡 핵심 답변
외주 프로젝트는 개발사에 맡기면 알아서 다 해주는 방식이 아니다. 요구사항을 명확히 전달하고, 중간 산출물을 확인하고, 완성 후 검수를 하는 역할이 클라이언트에게 있다. 이것이 잘 안 되면 완성 후 기대와 전혀 다른 결과물을 받는 가장 흔한 원인이 된다.
중간 확인은 기술 완성도를 평가하는 것이 아니라 처음 요청한 기능이 맞게 동작하는지를 직접 시나리오로 테스트하는 것이다. '이 화면에서 이 버튼을 누르면 이렇게 되어야 한다'는 기대를 기준으로, 실제로 그렇게 동작하는지 직접 사용해보는 것이 가장 현실적인 방법이다. 기능 단위 버그 수정은 개발사 역할이지만, 전체 시나리오가 맞는지 확인하는 것은 클라이언트 역할이다.
최종 검수는 처음 요청한 기능 목록 전체를 실제 사용 시나리오로 테스트하고, 계약에 명시된 범위가 모두 구현되었는지 확인하는 것이다. 검수 완료의 기준은 '기대한 기능이 모두 동작하는가'이지 '완벽한가'가 아니다. 검수 중 나온 버그는 하자보수 기간 내에 개발사가 수정하고, 새로 추가하고 싶은 기능은 유지보수 계약으로 이어가면 된다. 위시켓에서 진행했다면 에스크로 대금은 검수 후 클라이언트가 승인해야 지급되므로, 검수를 꼼꼼히 한 뒤 승인하면 된다.

▶ 개발사에 '중간 확인을 어떻게 진행할 건지' 일정과 방식을 먼저 물어본다
▶ 처음 작성한 기능 목록을 꺼내서, 각 기능이 실제로 동작하는지 하나씩 직접 사용해보고 안 되는 항목을 메모한다
