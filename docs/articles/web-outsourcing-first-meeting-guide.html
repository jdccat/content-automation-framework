외주 개발 첫 미팅, 비개발자가 준비할 3가지

💡 핵심 요약

완성된 기획서 없이도 첫 미팅은 할 수 있습니다. 업체가 견적을 내려면 화면 목록, 화면별 기능, 서버 통신 여부 이 세 가지가 메모 수준으로 있으면 충분하거든요. 미팅에서는 기술 스펙이 아니라 유사 경험·소통 방식·유지보수 가능 여부 세 가지를 확인하면 됩니다. 비교 기준을 미리 정해두면 견적이 다른 업체들을 같은 조건으로 놓고 판단할 수 있습니다.

웹서비스를 외주로 만들기로 했습니다. 기능도 대충 정리했고, 업체 몇 곳에 연락을 해볼 생각입니다. 그런데 막상 미팅을 잡으려니 멈칫하게 되죠. 기획서가 있어야 하는 건 아닌지, 업체에 뭘 어떻게 설명해야 하는지, 개발을 모르는 내가 질문은 할 수 있는 건지.

결론부터 말하면, 완성된 기획서 없이도 첫 미팅은 할 수 있습니다. 필요한 건 기획서가 아니라 '우리가 뭘 만들고 싶은지'를 메모 수준으로라도 정리한 목록이에요. 그리고 미팅에서 확인해야 할 것도 기술이 아닙니다. 기능 목록을 정리하는 법, 미팅에서 물어볼 질문, 미팅 후 업체를 비교하는 기준까지 순서대로 짚어보겠습니다.

기능 목록, 어떻게 정리해야 견적이 나올까?

업체가 견적을 내려면 최소한 세 가지 정보가 있어야 합니다. 어떤 화면이 필요한지, 각 화면에서 뭘 할 수 있는지, 서버와 통신이 필요한지 여부입니다. 와이어프레임이나 기획서까지 필요한 건 아니에요. 메모지에 쓴 수준이라도 이 세 가지가 있으면 파트너가 범위를 잡을 수 있습니다. 정리하는 가장 쉬운 방법은 서비스를 실제로 쓸 사람의 동선을 따라가는 겁니다.

단계별 정리법:

  1. 사용자 시나리오 쓰기 — "회원가입을 하고, 상품을 검색하고, 장바구니에 담고, 결제한다" 이 정도면 됩니다. 사용자가 들어와서 나가기까지의 흐름을 한 줄로 써봅니다
  2. 화면 나열 — 시나리오에서 화면이 바뀌는 지점을 찍습니다. 로그인 화면, 메인 화면, 상품 상세 화면, 결제 화면 같은 식으로요
  3. 화면별 기능 적기 — 각 화면에서 할 수 있는 일을 나열합니다. "상품 검색", "필터 적용", "리뷰 보기" 같은 단어 수준이면 충분해요
  4. 서버 통신 여부 표시 — 회원 데이터 저장, 결제 연동처럼 서버와 주고받아야 하는 기능에 체크합니다. 잘 모르겠으면 비워둬도 됩니다. 업체가 미팅에서 확인해줍니다

여기서 중요한 건 완성도가 아닙니다. 모든 업체에 같은 목록을 보내는 것이 핵심이에요. 업체마다 다른 내용을 설명하면, 나중에 견적이 나와도 같은 조건으로 비교할 수가 없거든요.

정리한 내용, 어떻게 전달하고 미팅에서 뭘 확인할까?

기능 목록은 이메일이나 카카오톡으로 보내면 됩니다. 엑셀에 화면명과 기능을 정리한 표 하나면 충분하고, 메모 앱에 적은 걸 그대로 공유해도 괜찮아요. 별도의 문서 양식은 필요 없습니다.

전달보다 중요한 건 첫 미팅에서 무엇을 확인하느냐입니다. 개발을 모르는 담당자가 기술 스택이나 아키텍처를 물어봐야 할 필요는 없어요. 답변을 들어도 판단할 수 없기 때문입니다. 대신, 이 세 가지를 물어보면 됩니다.

미팅 후, 업체를 비교할 때 기준은?

여러 업체와 미팅을 마치면 새로운 문제가 생깁니다. 업체마다 답변이 달라서, 누가 더 나은지 비교할 기준이 없다는 거죠. 이때 금액만 보고 판단하면 후회할 확률이 높습니다.

미팅에서 확인한 세 가지 항목을 업체별로 동일하게 기록해보세요. 금액 외의 비교가 가능해집니다.

비교할 때 하나 더. 견적 금액이 다르다면, 포함 범위가 같은지 먼저 확인해야 합니다. 기획 포함 여부, 디자인 수준, 관리자 페이지 유무가 다르면 금액 비교 자체가 성립하지 않아요. 비싼 견적이 바가지가 아니라 더 넓은 범위를 포함하고 있는 경우가 많습니다.

견적이 합리적인지 어떻게 알 수 있을까?

웹서비스 외주 견적이 합리적인지 판단하려면 비슷한 프로젝트의 예산 범위를 알고 있어야 합니다. 처음 외주를 맡기는 입장에서 이 감각이 없는 건 당연해요.

2025년 위시켓에 등록된 웹서비스 신규 개발 프로젝트를 보면, 기능 범위에 따라 예산이 크게 달라집니다.

같은 '웹서비스 신규 개발'인데 800만 원에서 2,500만 원까지, 약 3배 차이가 납니다. 이건 업체의 바가지가 아니라 기능 범위의 차이예요. 내 프로젝트의 기능 목록이 어느 규모에 가까운지를 먼저 가늠해보면, 받은 견적이 범위 안에 있는지 판단할 수 있습니다.

그래도 판단이 어렵다면 두 가지 방법이 있습니다.

  • 같은 범위의 견적을 3곳 이상 받아보기 — 기능 목록이 동일하면 견적 간 편차가 곧 시장 범위가 됩니다. 극단적으로 높거나 낮은 곳을 제외하면 중간 범위가 기준점이 돼요
  • 위시켓 매니저에게 적정 금액 상담 받기 — 위시켓에 프로젝트를 등록하면 매니저가 9만 건 이상의 프로젝트 상담 경험을 바탕으로 적정 금액과 기간을 안내합니다. 프로젝트 등록과 매니저 상담은 무료입니다

기능 목록을 정리했고, 미팅에서 확인할 질문도 알게 됐고, 업체를 비교할 기준도 생겼습니다. 개발을 몰라도 외주는 시작할 수 있어요. 기술을 아는 게 아니라 같은 기준으로 비교할 수 있는 상태를 만드는 것, 그게 첫 미팅 준비의 전부입니다.

자주 묻는 질문

Q.기획서 없이 외주 업체에 연락해도 되나요?
됩니다. 소규모·중규모 웹서비스 외주에서 와이어프레임이나 RFP를 요구하는 업체는 드뭅니다. 화면 목록과 각 화면별 기능을 메모 수준으로 정리한 것만 있어도 업체가 범위를 파악하고 견적을 낼 수 있습니다. 오히려 기획이 완성되길 기다리다 첫 미팅 자체를 미루는 것이 더 큰 시간 손실입니다. 기능 목록이 구체적일수록 견적이 정확해진다는 점만 인지하면 됩니다.
Q.업체마다 견적 범위가 달라서 금액을 비교할 수가 없습니다. 어떻게 해야 하나요?
견적을 비교하기 전에 포함 범위를 먼저 맞춰야 합니다. 기획 포함 여부, 디자인 수준, 관리자 페이지 유무, API 연동 수가 다르면 금액 비교 자체가 성립하지 않습니다. 같은 범위로 정리한 뒤에도 금액 차이가 크다면 투입 인원 구성을 직접 물어보는 것이 다음 단계입니다. 비싼 견적이 바가지가 아니라 더 넓은 범위를 포함하고 있는 경우가 대부분이에요.
Q.유지보수를 처음 계약에 포함해야 하나요?
현실에서는 최초 계약에 유지보수를 포함하는 경우가 거의 없습니다. 런칭 후 무엇이 필요할지 사전에 확정하기 어렵기 때문입니다. 미팅에서 확인해야 할 것은 유지보수 포함이 아니라 '런칭 후에 대응이 가능한지 여부'입니다. 대응 가능하다면 런칭 후 별도 계약으로 진행하면 됩니다. 유지보수와 하자보수는 다릅니다. 하자보수는 개발사 귀책 버그 수정으로 통상 3~6개월 무상 제공되는 별도 항목입니다.
Q.여러 업체와 미팅할 때 매번 같은 내용을 설명해야 하나요?
미팅 전에 기능 목록을 문서 하나로 정리해두면 반복 설명을 줄일 수 있습니다. 모든 업체에 같은 목록을 보내는 것이 중요한데, 업체마다 다른 내용을 설명하면 나중에 견적이 나와도 같은 조건으로 비교할 수가 없기 때문입니다. 엑셀 한 장이나 메모 앱에 적은 것을 이메일로 보내도 충분하고, 별도의 문서 양식은 필요 없습니다.
