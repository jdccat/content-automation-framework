프리랜서 웹사이트 외주, 계약에서 납품까지 4단계 관리 체크리스트

2026-04-07

💡 핵심 요약

프리랜서에게 웹사이트를 맡길 때 문제가 생기는 건 대부분 프리랜서 실력보다 관리 구조가 없어서입니다. 에이전시가 내부에서 처리하는 PM 역할을 클라이언트가 직접 맡아야 하거든요. 계약 전에 개발 범위를 기능 단위로 명세하고, 진행 중에는 주간 단위로 중간 산출물을 확인하고, 납품 후에는 계약서 기준으로 검수하는 루틴을 잡아두면 납품 단계에서 한꺼번에 문제가 터지는 상황을 막을 수 있습니다.

프리랜서에게 웹사이트 개발을 맡기기로 했습니다. 개발비가 에이전시보다 낮다는 건 알겠는데, 막상 시작하려니 어디서부터 뭘 해야 하는지가 안 보여요. 계약서에 뭘 넣어야 하는지, 개발 중간에 뭘 확인해야 하는지, 납품을 받으면 뭘 어떻게 검수하는지. 하나도 정해진 게 없는 상태에서 출발하는 거죠.

프리랜서 외주에서 문제가 생기는 대부분의 경우는 프리랜서의 실력 때문이 아닙니다. 관리 구조가 없는 상태에서 시작했기 때문이에요. 에이전시는 내부에 PM이 있고, 보고 체계가 있고, 계약서 양식이 잡혀 있습니다. 프리랜서 방식은 그 역할을 클라이언트가 직접 맡아야 해요. 위시켓에서 프리랜서 방식 프로젝트를 수백 건 봐온 경험에서 말하면, 계약 전에 구조를 잡아둔 프로젝트와 그렇지 않은 프로젝트는 진행 과정에서 분명한 차이가 납니다.

프리랜서를 어떻게 찾느냐부터 다릅니다

프리랜서를 연결하는 경로는 크게 두 가지이고, 같은 프리랜서라도 경로에 따라 계약 안전성과 관리 지원 구조가 완전히 달라집니다. SNS나 지인 소개로 직접 찾는 방식과 IT 아웃소싱 플랫폼을 경유하는 방식이에요.

프리랜서를 어떻게 찾느냐부터 다릅니다 — 항목 · 직접 연결 (SNS·지인 소개) · 플랫폼 경유 (위시켓 등)
항목직접 연결 (SNS·지인 소개)플랫폼 경유 (위시켓 등)
계약서직접 작성하거나 협의법무법인 검토 표준계약서 제공
대금 보호직접 관리 (선금 리스크)에스크로 방식 대금보호
분쟁 발생 시당사자 간 해결매니저 조율 가능
프리랜서 검증포트폴리오 직접 확인이전 프로젝트 평가 참고 가능
관리 지원없음매니저가 견적 상담·계약까지 동반

직접 연결이 무조건 나쁜 건 아닙니다. 이전에 함께 일해본 프리랜서라면 신뢰가 이미 쌓여 있으니까요. 다만 처음 만나는 프리랜서와 직접 연결하는 경우라면, 계약서 협의부터 대금 관리까지 전부 혼자 해야 합니다. 이 부담이 감당되는지를 먼저 판단하는 게 첫 단계예요.

프리랜서 연결 경로가 고민된다면 매니저와 먼저 상담해보세요.

무료 상담하기

계약 전에 확인해야 할 것들

프리랜서를 찾았다면 바로 계약서를 쓰기보다, 먼저 확인해야 하는 항목이 있습니다. 여기서 빠뜨리는 항목이 나중에 분쟁의 원인이 돼요.

✅ 프리랜서 계약 전 체크리스트
  • [ ] 유사 프로젝트 경험이 있는가? (포트폴리오에서 비슷한 유형·규모 확인)
  • [ ] 개발 범위가 구체적으로 명세되어 있는가? (기능 목록, 페이지 수, 반응형 여부)
  • [ ] 개발 완료 기준이 정의되어 있는가? ("완성"이 무엇인지 합의)
  • [ ] 하자보수 기간과 범위가 계약서에 있는가?
  • [ ] 소스코드·지재권 귀속이 계약서에 명시되어 있는가?
  • [ ] 중간 산출물 확인 일정이 합의되어 있는가?
  • [ ] 대금 지급 조건과 시점이 명확한가? (착수금·중도금·잔금 비율)

이 중에서 가장 많이 빠뜨리는 항목이 개발 범위 명세입니다. "웹사이트 만들어주세요"라는 구두 합의로 시작하면, 프리랜서가 생각하는 범위와 클라이언트가 기대하는 범위가 다를 수밖에 없어요. "메인 페이지 1개 + 서브 페이지 5개 + 문의 폼 + 관리자 페이지"처럼 기능 단위로 쪼개서 명세를 잡아야 합니다.

2025년 위시켓에 등록된 웹사이트 프로젝트 약 220건을 보면, 같은 '웹사이트 개발'이라도 범위에 따라 예산이 300만 원에서 2,500만 원까지 벌어집니다. 웹빌더 기반 정보 제공 사이트(300만 원, 14일)와 브랜드 이커머스 풀 구축(2,500만 원, 60일)은 이름만 같을 뿐 완전히 다른 프로젝트예요. 범위를 명세하지 않으면 프리랜서는 300만 원짜리를 생각하고, 클라이언트는 2,500만 원짜리를 기대하는 상황이 생깁니다.

플랫폼 경유라면 표준계약서에 하자보수·지재권·대금보호 항목이 이미 포함되어 있고, 매니저와 함께 내 프로젝트에 맞게 조정할 수 있어요. 직접 연결 방식이라면 이 항목들을 프리랜서와 별도로 협의해서 계약서에 넣어야 합니다.

개발 중에 확인하지 않으면 납품 때 한꺼번에 터집니다

계약이 끝나면 프리랜서가 개발에 들어갑니다. 여기서 가장 흔한 실수는 "맡겼으니 완성될 때까지 기다리는 것"이에요. 에이전시와 달리 프리랜서 방식에서는 중간 확인을 클라이언트가 직접 챙겨야 합니다.

위시켓 데이터 기준, 웹사이트 신규 개발 프로젝트의 기간 중위값은 30일이고 상위 25%는 60일 이상입니다. 30일짜리 프로젝트라면 주간 확인이 4번, 60일이면 8번이에요. 이 확인 기회를 한 번도 쓰지 않고 납품일까지 기다리면, 문제가 생겨도 돌이킬 시간이 없습니다.

주간 단위로 중간 산출물을 확인하는 루틴을 만드세요. 거창한 보고서가 아니어도 돼요. 이번 주에 뭘 했는지, 다음 주에 뭘 할 건지, 막히는 부분이 있는지. 이 세 가지만 공유해도 충분합니다.

  • 1~2주차: 화면 설계(와이어프레임)나 디자인 시안 확인. 이 단계에서 방향이 어긋나면 적은 비용으로 수정할 수 있습니다
  • 3~4주차: 주요 기능이 동작하는 중간 빌드 확인. 직접 클릭해보고 의도한 흐름대로 동작하는지 체크합니다
  • 마지막 주: 전체 기능 시나리오 테스트. 계약서에 명시한 기능 목록을 기준으로 하나씩 확인합니다

이상 신호가 보일 때 바로 짚어두지 않으면, 납품 단계에서 "이건 제가 들은 것과 다릅니다" vs "저는 이렇게 합의했다고 알고 있습니다"가 됩니다. 문서로 남기지 않은 구두 합의는 어느 쪽이든 증명이 안 돼요.

📌 위시켓 매니저 Tip
플랫폼 경유로 진행 중이라면 중간에 문제가 생겼을 때 매니저에게 상황을 공유하고 조율을 요청할 수 있습니다. 직접 연결 방식에서는 이 역할도 클라이언트 본인의 몫입니다.

납품 후 수정 요청, 무상과 유상의 기준

개발이 완료되면 최종 납품물을 검수해야 합니다. 검수 기준은 계약서에 명시한 기능 목록과 개발 완료 기준이에요. 계약 전에 이걸 구체적으로 잡아두는 게 중요한 이유가 여기에 있습니다.

납품 후 수정이 필요할 때 가장 자주 분쟁이 생기는 지점은 하자보수와 유지보수의 구분이에요.

납품 후 수정 요청, 무상과 유상의 기준 — 구분 · 하자보수 · 유지보수
구분하자보수유지보수
대상계약 범위 내 버그·오류기능 추가, 디자인 변경, 운영 중 수정
비용무상 (프리랜서 귀책)유상 (별도 계약)
기간계약서에 명시된 기간 내별도 유지보수 계약 기간
예시결제 버튼이 동작하지 않음새로운 결제 수단 추가

"이건 버그 아닌가요?"와 "이건 추가 기능입니다"의 경계에서 다툼이 생깁니다. 이 구분을 명확하게 하려면 계약 단계에서 개발 완료 기준을 기능 단위로 정의해두는 수밖에 없어요. "정상 동작"의 기준이 문서로 있어야 하자인지 아닌지를 판단할 수 있습니다.

위시켓 데이터를 보면 웹사이트 유지보수 프로젝트의 예산 중위값은 300만 원입니다. 신규 개발 중위값(725만 원)과는 별개의 비용이에요. 납품 후에도 기능 추가나 운영 수정이 필요하다면 이 비용까지 미리 잡아두는 게 현실적입니다.

한 가지 더. 프리랜서와의 계약이 끝나면 유지보수를 맡길 사람이 없어지는 경우가 많습니다. 6개월, 1년 뒤에 수정이 필요할 때 같은 프리랜서에게 맡길 수 있는지, 아니면 다른 사람에게 인수인계가 가능한 상태인지를 미리 확인해두세요. 소스코드와 인수인계 문서가 계약에 포함되어야 하는 이유입니다.

프리랜서 방식이 나쁜 방식은 아닙니다. 예산이 제한적이고, 프로젝트 규모가 크지 않고, 관리에 시간을 투자할 수 있다면 충분히 좋은 선택이에요. 다만 에이전시가 해주는 관리 역할을 클라이언트가 직접 맡아야 한다는 점을 알고 시작하는 것과 모르고 시작하는 것은 결과가 다릅니다.

이 관리 구조를 혼자 잡기 어렵다면, 프로젝트를 위시켓에 등록하는 것도 방법입니다. 매니저가 요구사항 정리부터 계약까지 함께하고, 대금보호 구조가 갖춰진 상태에서 프리랜서와 일할 수 있어요. 프로젝트 등록과 매니저 상담은 무료이니 기획이 아직 정리되지 않은 상태라도 먼저 상담부터 시작할 수 있습니다.

자주 묻는 질문

Q.프리랜서와 직접 계약할 때 표준계약서가 없으면 어떻게 해야 하나요?
직접 연결 방식이라면 표준계약서 없이 시작하는 경우가 많습니다. 이때는 최소한 개발 범위·완료 기준·하자보수 기간·소스코드 귀속 조항 4가지를 계약서에 명시해야 합니다. 법무법인 검토를 거친 표준계약서가 필요하다면 위시켓 같은 플랫폼을 통해 연결하면 별도 비용 없이 제공받을 수 있습니다. 구두 합의는 분쟁 시 증명이 안 되므로, 작은 변경 사항도 채팅·이메일로 남겨두는 습관이 중요합니다.
Q.개발 중간에 요구사항이 바뀌면 추가 비용을 내야 하나요?
계약 범위 안에서의 수정이면 추가 비용 없이 반영할 수 있지만, 계약에 없던 기능을 새로 요청하는 경우에는 별도 협의가 필요합니다. 이 구분이 명확하지 않으면 "이건 당연히 포함 아니었나요?"라는 갈등이 생깁니다. 계약 전에 기능 목록을 구체적으로 명세해두는 것이 이 갈등을 예방하는 가장 확실한 방법입니다. 범위 변경이 필요할 때는 구두보다 서면으로 합의하고, 추가 비용과 일정 영향을 함께 확인하세요.
Q.프리랜서가 중간에 연락이 두절되면 대금을 어떻게 보호하나요?
직접 연결 방식에서는 이미 지급한 선금을 돌려받기가 현실적으로 어렵습니다. 리스크를 줄이려면 착수금 비율을 30% 이하로 낮추고, 중도금을 산출물 확인 후 지급하는 조건으로 계약하는 것이 좋습니다. 플랫폼을 통해 연결하면 에스크로 방식의 대금보호 구조가 있어 프로젝트 완료 전까지 대금이 보관됩니다. 연락 두절 자체를 완전히 막을 수는 없지만 대금 손실 리스크는 경로 선택으로 줄일 수 있습니다.
Q.납품 후 6개월이 지나서 버그가 발견되면 무상 수정을 받을 수 있나요?
하자보수 기간이 지난 후 발견된 버그는 무상 수정 대상이 아닙니다. 일반적으로 하자보수 기간은 계약서에 명시된 1~3개월이 기준이고, 이 기간 안에 발견된 개발사 귀책 버그만 무상으로 처리됩니다. 6개월 후 수정은 별도 유지보수 계약이 필요한 유상 작업입니다. 납품 직후 계약 기간 내에 전체 기능을 꼼꼼히 테스트하고, 이상이 있으면 하자보수 기간 안에 요청하는 것이 중요합니다.
